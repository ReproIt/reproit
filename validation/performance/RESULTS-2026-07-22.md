# CLI hot-path results, 2026-07-22

This comparison used the fixed workload matrix in `run-cli-hotpaths.sh` on the
same otherwise idle machine. Each entry is the median of seven samples. The
baseline is commit `50a392b236fa6e9027942349ec5c0d018731b4cd`, where the
benchmark suite was introduced. The candidate is the 1.0 working tree based on
`e6540afe5007ad45fc75bb8fb2acaf93d467b653` and containing this results file.

Environment:

- Apple M1 Ultra, 128 GiB memory;
- macOS 26.1 (25B78);
- rustc 1.96.0 and cargo 1.96.0;
- release profile with the `perf-bench` feature;
- identical commands, workload sizes, iteration counts, and sample counts.

## Correctness and resource tradeoffs

Frontier and batch selection now analyze strongly connected components and
dominator reach. That additional deterministic guidance prefers states that
unlock more of the reachable graph instead of breaking equally visited targets
only by depth and name. It is the source of the frontier and batch increase.
The candidate remains bounded at 20,000 states and 100,000 edges for component
analysis, and at 1,024 states and 16,384 edges for dominator analysis.

The first candidate measurement exposed an unacceptable implementation of that
tradeoff: the 1,000-state frontier used 182.7 MB of allocation and took 58.3 ms.
Replacing full ordered dominator sets with an immediate-dominator tree reduced
the same operation to 2.63 MB and 1.95 ms while preserving the guidance tests.
The 10,000-state case intentionally skips dominator analysis at its documented
bound and completes in 18.13 ms.

Log parsing is linear in input bytes. Its 100 MiB workload allocates about 2.13
GB cumulatively while retaining a peak of about 115 MB because the benchmark
materializes the parsed production structures. It completes in 1.54 seconds on
this machine. This is not an input limit: production capture independently
bounds subprocess output and retained evidence. The benchmark records the cost
so future changes can separate parser work from those boundary limits.

No timing in this document is a universal hardware guarantee. A release
comparison is valid only when both revisions use the same machine, toolchain,
power state, and workload matrix.
